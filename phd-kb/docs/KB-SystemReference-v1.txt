## Arquitectura General — TRES PIPELINES

El sistema tiene **tres pipelines independientes** que alimentan el mismo Google Sheet:

```
┌─────────────────────────────────────────────────────────────────────┐
│                        GOOGLE SHEET                                 │
│                                                                     │
│  ┌──────────────┐  ┌───────────────┐  ┌────────────────────┐       │
│  │ NewsLog      │  │ AcademicQueue │  │ NewsResults        │       │
│  │ 1,041 entries│  │ Papers crudos │  │ Noticias crudas    │       │
│  │ KB CURADO    │  │ sin evaluar   │  │ sin evaluar        │       │
│  └──────┬───────┘  └──────┬────────┘  └──────┬─────────────┘       │
│         │                 │                   │                     │
│  ┌──────┴───────┐  ┌──────┴────────┐  ┌──────┴─────────────┐       │
│  │ Queries      │  │ Queries       │  │ Queries            │       │
│  │ (q_fullweb)  │  │ (q_openalex…) │  │ (q_news)           │       │
│  └──────┬───────┘  └──────┬────────┘  └──────┬─────────────┘       │
│         │                 │                   │                     │
│  ┌──────┴───────┐  ┌──────┴────────┐  ┌──────┴─────────────┐       │
│  │ Meta         │  │ Meta          │  │ Meta               │       │
│  │ (timestamps) │  │ (timestamps)  │  │ (timestamps)       │       │
│  └──────────────┘  └───────────────┘  └────────────────────┘       │
└─────────────────────────────────────────────────────────────────────┘
         ▲                   ▲                   ▲
         │                   │                   │

  PIPELINE 1             PIPELINE 2          PIPELINE 3
  ─────────              ──────────          ──────────
  SKILL (Claude)         ACADÉMICO           NOTICIAS
  ─────────              ──────────          ──────────
  Trigger: usuario       Trigger: cron       Trigger: cron
  dice "news" o "nd"     runIncrementalScan  runNewsAPI
                         startDeepSweep      (lunes 6am)

  Método:                Método:             Método:
  Claude web_search      OpenAlex.gs         NewsAPI.org
  + evaluación AI        SemanticScholar.gs  (newsapi.org/v2)
  + 3-párrafo análisis   ArXiv.gs
                         CORE.gs

  Acceso al Sheet:       Acceso al Sheet:    Acceso al Sheet:
  SHEET_API (web app)    SpreadsheetApp      SpreadsheetApp
  via HTTP/curl          (directo)           (directo)

  Output:                Output:             Output:
  → NewsLog             → AcademicQueue     → NewsResults
    (curado, evaluado)    (papers crudos)     (noticias crudas)
```

**NewsLog es la pestaña más importante del sistema.** Es el Knowledge Base curado de la tesis: 1,041 entradas, cada una con análisis de relevancia de 3 párrafos, clasificación por capa, importancia, y criterios evaluativos. Los otros dos pipelines (académico y noticias) producen resultados crudos que eventualmente se revisan y, si son relevantes, se promueven a NewsLog.

## index.html — Knowledge Base Webapp

**Repositorio:** https://github.com/aauml/thesis
**URL pública:** https://aauml.github.io/thesis/
**Deploy:** Automático — cada push a `main` actualiza GitHub Pages en ~1-2 min.

### Arquitectura

```
GitHub Pages (index.html)
       │
       │  fetch() — CORS OK (access-control-allow-origin: *)
       ▼
Apps Script Web App (WebApp.gs, deployment v28)
       │
       ▼
Google Sheet → pestaña NewsLog
```

Es un archivo HTML/CSS/JS autocontenido sin dependencias externas ni build process. Carga **todos los datos de una vez** (`limit=2000&offset=0`), los cachea en `_cache`, y hace todo el filtrado, ordenamiento y paginación en el browser (client-side). Esto es necesario porque el API tiene limitaciones importantes (ver tabla abajo).

### Limitaciones del API — verificadas con curl (2026-03-08)

| Parámetro | Server-side | Notas |
|---|---|---|
| `limit` | ✅ | Funciona. Máx testeado: 2000 (devuelve todos los rows) |
| `offset` | ✅ | Funciona para paginación |
| `importance` | ✅ parcial | Filtra rows correctamente pero `total` **siempre devuelve 1041** (total global, nunca filtrado) |
| `capa` | ✅ parcial | Igual — filtra rows pero `total` no refleja el filtro |
| `search` | ❌ | Ignorado. Se hace client-side |
| `sort` | ❌ | Ignorado. Se hace client-side |
| `capa_detail` | ❌ | Ignorado. Se hace client-side |
| `callback` (JSONP) | ❌ | API devuelve JSON plano; ignorar parámetro `callback`. Usar `fetch()` |

**Formato de respuesta:** `{ "rows": [...], "total": 1041 }` — no incluye `total_pages`, `has_next`, `has_prev`, ni `stats`.

**Valores de campos a tener en cuenta:**
- `starred`: el Sheet devuelve el **entero** `1`, no el string `"1"`. Comparar con `==` (loose equality).
- `capa`: inconsistencia de acentos en ~22 entradas (ver GAP 2 abajo). La webapp usa `normStr()` para comparación accent-insensitive.

### Funciones principales

| Función | Qué hace |
|---|---|
| `loadAllRows()` | Fetch una vez, cachea en `_cache`. Calcula stats de importancia. |
| `fetchData()` | `loadAllRows()` → filter → sort → paginate → render. Todo client-side. |
| `normStr(s)` | Normaliza acentos para comparaciones robustas (maneja Teorica/Teórica). |
| `refresh()` | Borra `_cache`, llama `fetchData()`. Fuerza re-fetch del Sheet. |
| `toggleStar()` | Escribe via `action=updateByUrl` (GET). Actualiza `_cache` y `state.rows` inmediatamente. |

### Historial de bugs y fixes

| Fecha | Bug | Causa | Fix |
|---|---|---|---|
| 2026-03-07 | Sin datos — "Loading…" indefinido | `buildUrl()` usaba `action=getPage`, que no existe en WebApp.gs v28 | Cambiar a `action=getAll` con `offset=(page-1)*limit` |
| 2026-03-07 | Sin datos con `action=getAll` | Webapp usaba JSONP (`<script>&callback=_cb_xyz`). API ignora `callback`, devuelve JSON plano. Browser ejecutaba `{"rows":...}` como JS → SyntaxError silencioso | Reemplazar `jsonp()` con `fetch()`. API tiene CORS habilitado (redirect 302→200, ambas respuestas con `access-control-allow-origin: *`) |
| 2026-03-08 | Filtros no funcionan, total siempre 1041, starred vacío | API siempre devuelve `total: 1041`; `search`/`sort`/`capa_detail` ignorados server-side; `starred` es int `1` no string `"1"`; filtro starred solo escaneaba 50 rows de la página actual | Cargar todos los datos (`limit=2000`), cachear, filtrar/ordenar/paginar todo client-side. Comparación starred con `==`. |

### Modificar la webapp

- **Agregar filtro nuevo:** añadir UI en HTML + filtro en `fetchData()` sección "── Filter ──" + reset en `clearAllFilters()` y `updateClearBtn()`
- **Nueva columna del NewsLog:** el API la devuelve automáticamente; acceder via `item.nombre_campo` en `renderCard()`
- **Agregar subcapas:** modificar `SUBCAPA_OPTIONS` (keys: `all`, `teorica`, `analitica`, `evaluativa`, `metodologica`)
- **Items por página:** cambiar constante `PAGE_LIMIT` (actualmente 50)
- **Si el Apps Script cambia:** actualizar `API_URL`; verificar que devuelva `{ rows: [...], total: N }` con CORS headers

## Apps Script — Archivos

### `appsscript.json` — Manifiesto del proyecto
**Para qué sirve:** Configuración base del proyecto. Define el runtime, permisos OAuth, y zona horaria.

**Campos importantes:**
- `timeZone`: `America/Los_Angeles`
- `runtimeVersion`: `V8` (JavaScript moderno)
- `oauthScopes`: `spreadsheets` + `external_request`
- `webapp.access`: `ANYONE_ANONYMOUS` — necesario para que el SKILL pueda llamar a la SHEET_API desde fuera

**Cuándo modificar:** Solo si se agregan scopes nuevos (Drive, Gmail, etc.) o cambia la zona horaria.

**Troubleshooting:** Si el script pide reautorizar, probablemente se modificó un scope aquí.

### `AcademicOrchestrator.gs` — Orquestador académico

**Función única: pipeline académico.** Lee las queries de la pestaña Queries, las distribuye entre los 4 proveedores académicos, guarda los resultados en AcademicQueue, y maneja el sistema de batches con triggers automáticos.

> **Nota:** La API del web app (doGet/doPost) ya NO está en AcademicOrchestrator.gs — está en `WebApp.gs` (ver sección siguiente). Si necesitas agregar acciones a la SHEET_API, edita WebApp.gs.

**Objeto CONFIG:**

| Parámetro | Valor | Para qué |
|---|---|---|
| `RESULTS_PER_QUERY` | 3 | Papers máximos por query por proveedor |
| `BATCH_SIZE` | 15 | Queries procesadas por ejecución |
| `BATCH_GAP_MS` | 70,000 ms | Espera entre batches (evita timeout de Apps Script) |
| `QUERY_TYPES` | `['scholar', 'topic']` | Tipos de query que procesa |
| `QUEUE_TAB` | `'AcademicQueue'` | Pestaña donde escribe |

**Funciones del orquestador:**
- `runIncrementalScan()` — Entry point del trigger regular. Procesa el siguiente batch.
- `startDeepSweep()` — Barrido histórico desde 2020. Resetea sweep_index.
- `_cleanBatchTriggers()` — Limpia triggers acumulados.

**Proveedores mapeados:**
- `q_openalex` → `searchOpenAlex()` en OpenAlex.gs
- `q_semantic` → `searchSemanticScholar()` en SemanticScholar.gs
- `q_arxiv` → `searchArXiv()` en ArXiv.gs
- `q_core` → `searchCORE()` en CORE.gs

**Cuándo modificar:**
- Cambiar el tamaño del batch académico: editar CONFIG
- Agregar un proveedor académico: entrada en PROVIDERS + nuevo .gs en **AcademicOrchestrator.gs**

**Troubleshooting:**
- "too many triggers": correr `_cleanBatchTriggers()`
- Scan se detiene: revisar `sweep_index` en Script Properties

### `WebApp.gs` — API del web app (SHEET_API) ✅ NUEVO en v28

**Para qué sirve:** Contiene las funciones `doGet()` y `doPost()` que exponen la SHEET_API. Cuando el SKILL corre `curl` contra la URL del deployment, es WebApp.gs quien procesa la petición.

**Historial:** Esta lógica existía en versiones antiguas de AcademicOrchestrator.gs (hasta v27). Fue reconstruida en un archivo separado WebApp.gs y desplegada como v28 el 7 Mar 2026.

**Acciones GET (doGet):**

| Action | Lee | Descripción |
|---|---|---|
| `getMeta` | Meta | Timestamps y metadatos del sistema |
| `getAll` | NewsLog | Todas las entradas (soporta filtros: `importance`, `folder`, `action_tag`, `limit`) |
| `getUrls` | NewsLog | Solo URLs — para deduplicación rápida |
| `getQueries` | Queries | Queries activas (filtra por `active=yes`) |
| `getNewsResults` | NewsResults | Resultados crudos del pipeline de noticias (soporta `status`, `limit`) |

**Acciones POST (doPost):**

| Action | Escribe | Descripción |
|---|---|---|
| `append` | NewsLog | Agrega filas nuevas (dedup por URL) |
| `updateRow` | NewsLog | Actualiza campos de una fila existente (por `id`) |
| `updateMeta` | Meta | Actualiza timestamps y contadores |
| `updateNewsRow` | NewsResults | Actualiza `status` y `notes` de un resultado de noticias |

**Cuándo modificar:**
- Agregar una nueva acción a la SHEET_API: agregar case en doGet o doPost
- Si se cambian columnas en NewsLog: actualizar la lógica de `append` y `getAll`
- Si se cambian columnas en NewsResults: actualizar `getNewsResults` y `updateNewsRow`

**IMPORTANTE:** Cada vez que se modifica WebApp.gs, hay que **redesplegar** el web app para que los cambios se reflejen en la SHEET_API. Ir a Deploy → Manage deployments → editar → New version → Deploy.

**Troubleshooting:**
- SKILL da error de API: verificar que el deployment esté activo (Deploy → Manage deployments). Deployment activo: v28
- API devuelve HTML en lugar de JSON: el deployment no está actualizado — redesplegar
- "Unknown action": la acción no está implementada en doGet/doPost de WebApp.gs

### `OpenAlex.gs` — Proveedor OpenAlex
**Pipeline:** Académico (llamado por AcademicOrchestrator.gs)
**API:** `https://api.openalex.org/works`
**Clave:** No requiere (gratuita, abierta)
**Rate limit:** Usa `mailto: thesis-scanner@academic.local` como cortesía

**Campos que devuelve (formato estandarizado):**
- `url`: landing page (con fallback a URL de OpenAlex)
- `title`, `authors` (primeros 3), `year`, `abstract` (reconstruido desde inverted index)
- `source_api`: `'OpenAlex'`

**Cuándo modificar:**
- Más de 3 autores: cambiar `.slice(0,3)`
- Más campos (DOI, venue, citations): agregar al `select` y al objeto de retorno

**Troubleshooting:**
- HTTP 429: rate limit. Agregar `Utilities.sleep()`.
- Abstract vacío: normal, no todos los papers tienen abstract.

---

### `SemanticScholar.gs` — Proveedor Semantic Scholar
**Pipeline:** Académico (llamado por AcademicOrchestrator.gs)
**API:** `https://api.semanticscholar.org/graph/v1/paper/search`
**Clave:** No requiere para uso básico. Fuerte en AI, NLP, CS.

**Cuándo modificar:** Filtrar por campo de estudio, ajustar rate limits.

**Troubleshooting:** HTTP 429 — el más restrictivo de los 4 proveedores. Reducir RESULTS_PER_QUERY o agregar delays.

### `ArXiv.gs` — Proveedor ArXiv
**Pipeline:** Académico (llamado por AcademicOrchestrator.gs)
**API:** `http://export.arxiv.org/api/query` (Atom XML, no JSON)
**Clave:** No requiere. Totalmente abierta.

**Cuándo modificar:** Filtrar por categoría (`cat:cs.AI`), ajustar rango de fechas.

**Troubleshooting:** Resultados con lag, errores de parsing XML silenciosos.

### `CORE.gs` — Proveedor CORE
**Pipeline:** Académico (llamado por AcademicOrchestrator.gs)
**API:** `https://api.core.ac.uk/v3/search/works`
**Clave:** Funciona sin clave (límites reducidos). Se puede agregar `CORE_API_KEY` en Script Properties para mayor cuota.
**Fuerte en:** Papers de universidades europeas, policy documents.

**Troubleshooting:** Más lento e inconsistente que los otros. Si da timeout frecuente, desactivar o reducir cuota.

## Google Sheet — Pestañas

### ★ `NewsLog` — Knowledge Base curado de la tesis (PESTAÑA PRINCIPAL)
**Estado:** Activa — **es la pestaña más importante del sistema**
**Entradas:** 1,041
**Escrita por:** Pipeline 1 (SKILL de Claude, via SHEET_API `action=append`)
**Leída por:** SKILL via `getAll`, `getUrls`, `updateRow`, `updateByUrl`, `deleteRow`

**Qué es:** La base de conocimiento curada de la tesis. Cada entrada es un recurso (artículo, paper, ruling, video, etc.) que fue encontrado por el SKILL mediante web search, evaluado por Claude con criterios de relevancia, importancia y clasificación por capas de la tesis, y almacenado con un análisis de 3 párrafos en el estilo de Gregory C. Allen.

**Columnas (21):**

| Col | Campo | Descripción |
|---|---|---|
| A | `id` | ID único (base64 de URL, primeros 16 chars) |
| B | `url` | URL del recurso |
| C | `title` | Título |
| D | `source` | Fuente con fecha (ej: "Internet Policy Review (Ben Crum, Aug 2025)") |
| E | `date_published` | Fecha de publicación del recurso |
| F | `date_found` | Cuándo lo encontró el SKILL |
| G | `content_type` | Tipo: `paper`, `article`, `report`, `blog`, `podcast`, `video`, `event`, `court_ruling`, `regulation`, `guidance`, `presentation`, `news`, `transcript`, `book`, `chapter`, `handbook` |
| H | `importance` | `ALTA` / `MEDIA` / `BAJA` — calibración: 1-3 ALTA, 5-10 MEDIA por scan |
| I | `capa` | Capa(s) de la tesis: `Teórica`, `Analítica`, `Evaluativa`, `Metodológica`, `Caso` |
| J | `capa_detail` | Detalle: ej. "Gobernanza IA comparada, Art. 9 ↔ MAP/MEASURE" |
| K | `evaluativa_criteria` | Criterios evaluativos: `Contestabilidad`, `Accountability`, `Equidad`, `Explicabilidad` |
| L | `action_tag` | `REFERENCE` (citable), `CONTEXT` (informa), `FOLLOW-UP` (monitorear), `URGENT` (inmediato) |
| M | `thesis_relevance` | **Campo más importante.** 3 párrafos: Contexto → Artículo → Tesis. Estilo Gregory Allen. Nunca truncar. |
| N | `scholar` | Nombre del scholar si aplica |
| O | `search_scope` | Qué búsqueda lo encontró (ej: "full scan", "nd: forensic") |
| P | `language` | `en`, `es`, etc. |
| Q | `run_id` | ID del scan: formato `YYYY-MM-DD-NNN` |
| R | `tier` | Confiabilidad: `1` = oficial/gobierno, `2` = académico/experto, `3` = general |
| S | `folder` | Organización manual: `01_Marco_Teorico`, `02_Metodologia`, `03_Corpus_Juridico`, `04_Estado_del_Arte` |
| T | `notes` | Notas adicionales |
| U | `starred` | Marcado manual |

**Relación con las capas de la tesis:**

La columna `capa` conecta cada entrada con la estructura de la tesis:

| Capa | Subcategorías en `capa_detail` |
|---|---|
| **Teórica** | Gobernanza IA comparada, Metodología derecho comparado, Derecho procesal y prueba electrónica, Contestabilidad algorítmica |
| **Analítica** | Art. 9 ↔ MAP/MEASURE, Art. 10 ↔ MAP/GOVERN, Art. 11 ↔ GOVERN, Art. 12 ↔ MEASURE, Art. 13 ↔ GOVERN/MAP, Art. 14 ↔ GOVERN/MANAGE, Art. 15 ↔ MEASURE/MANAGE |
| **Evaluativa** | Contestabilidad, Accountability, Equidad, Explicabilidad |
| **Metodológica** | Metodología de derecho comparado, marcos analíticos, diseño de investigación |
| **Caso** | Estudios de caso, implementaciones concretas, casos forenses, ejemplos jurisdiccionales |

**Cuándo modificar manualmente:**
- Cambiar `importance` después de leer el recurso
- Agregar `notes` con reflexiones
- Marcar `starred` para items prioritarios
- Mover a un `folder` temático
- Cambiar `action_tag` de FOLLOW-UP a REFERENCE si se decide citar

**Troubleshooting:**
- Si el SKILL no puede escribir: verificar que el deployment del web app esté activo (WebApp.gs, actualmente v28)
- Si hay duplicados: la API deduplica por URL, pero el `id` también ayuda
- Para buscar en el KB: usar el SKILL con "what did we find about [topic]?" o `action=getAll` via curl

## Flujo completo: cómo se conecta todo

### Cuando el usuario dice "news" o "nd":

```
1. SKILL lee config.txt → obtiene SHEET_API URL
2. curl SHEET_API?action=getMeta → obtiene last_search_date, total_searches
3. curl SHEET_API?action=getUrls → obtiene 1,041 URLs para dedup
4. curl SHEET_API?action=getQueries → obtiene 115 queries activas
5. SKILL ejecuta web_search por cada query relevante
6. Claude evalúa cada resultado: importancia, capa, thesis_relevance (3 párrafos)
7. curl SHEET_API action=append → escribe filas nuevas en NewsLog
8. curl SHEET_API action=updateMeta → actualiza last_search_date
```

### Cuando corre el trigger académico (runIncrementalScan):

```
1. AcademicOrchestrator.gs lee Queries tab (SpreadsheetApp directo)
2. Toma siguiente batch de 15 queries con q_openalex, q_semantic, etc.
3. Llama a cada proveedor (OpenAlex, SemanticScholar, ArXiv, CORE)
4. Escribe papers nuevos en AcademicQueue
5. Programa trigger para el siguiente batch
```

### Cuando corre el trigger NewsAPI (runNewsAPI):

```
1. NewsSearch.gs lee Queries tab → filas con q_news no vacío
2. Para cada query: llama a newsapi.org/v2/everything
3. Deduplica contra URLs conocidas (AcademicQueue + NewsResults)
4. Escribe artículos nuevos en NewsResults
5. Actualiza last_news_scan en Meta
```

### Cuando el SKILL usa "news api" (revisa NewsResults pendientes):

```
1. curl SHEET_API?action=getNewsResults&status=pending&limit=20
2. Claude evalúa cada artículo: relevancia para la tesis
3. Para artículos relevantes: evalúa capa, importancia, thesis_relevance
4. curl SHEET_API action=append → promueve a NewsLog
5. curl SHEET_API action=updateNewsRow → marca status=reviewed en NewsResults
```

### Gaps de coordinación encontrados

#### GAP 1: `q_news` casi vacío
**Problema:** Solo 1 de 115 queries tiene `q_news` poblado ("EU AI Act — Implementation" → "EU AI Act NIST RMF"). Las otras 108 queries activas no están siendo monitoreadas por NewsAPI.
**Impacto:** El pipeline de noticias (NewsSearch.gs) solo busca un término. Debería cubrir todos los topics activos.
**Solución:** Poblar `q_news` para las queries activas. Para scholars usar `"Nombre Completo" AI`. Para topics usar el término en inglés. Para government/media adaptar al contexto de noticias.

#### GAP 2: Capas con y sin acentos
**Problema:** Existen entradas con "Teórica" (721) y "Teorica" (11), "Analítica" (408) y "Analitica" (9), "Metodológica" (80) y "Metodologica" (2). Hay inconsistencia de acentos.
**Impacto:** Búsquedas y filtros por capa podrían perder entradas si filtran con/sin acento.
**Solución:** Normalizar las 22 entradas sin acento. Usar `updateByUrl` o editar directamente en el Sheet. El SKILL usa siempre las versiones con acento — asegurarse de que futuros appends sean consistentes.

#### ~~GAP 3: Capa "Caso" no documentada en el SKILL~~ ✅ RESUELTO en SKILL-KB-v4
**Problema original:** 71 entradas tienen capa "Caso" pero el SKILL-KB-v3 solo documentaba tres capas: Teórica, Analítica, Evaluativa.
**Resolución:** Capa "Caso" y "Metodológica" agregadas a SKILL-KB-v4 con definición y contexto de uso.

#### ~~GAP 4: Content types no documentados~~ ✅ RESUELTO en SKILL-KB-v4
